# Config-migration verification

End-to-end exercise of the 4-tool config-migration chain (gen-schema →
tdb get-snapshot → migrate.py → gen-transform) against a live etcd-backed `per`,
proving the **lockstep invariant**: the offline JSON design bench (migrate.py)
and the online nanopb plugin (gen-transform → dlopen'd by per's MigrateBulk)
produce **identical** results on a real config.

## What's here (tracked)

- `schema_v1.json` — p1/p2/p3 configs only (CounterConfig / ObserverConfig /
  IncrementerConfig), the starting shapes.
- `schema_v2.json` — adds P4Config **v1** (`{timeout_s, tag}`).
- `schema_v3.json` — P4Config **v2** (`{timeout_s, label, step, poll_ms, amount}`)
  — the consolidation target (folds in p1/p2/p3 fields, renames `tag→label`).
- `<node>_v1_to_v2.json` — the per-node transforms. SCAFFOLDED by `artheia
  gen-migration` (schema diff → auto from/to digest + add/remove + rename
  heuristic; `_review` notes flag the guesses), then human-edited (defaults,
  confirmed renames, custom hooks). E.g. `observer_v1_to_v2.json` is a rename,
  `counter_v1_to_v2.json` an add, `p4config_v1_to_v2.json` a rename + adds.
- `<node>_v1_to_v2.cc` — the plugins, generated by `artheia gen-transform`.
- `BUILD.bazel` — builds the plugin `.so` (self-contained: demo.pb.c compiled in,
  nanopb linked statically, NO runtime shared-lib deps — per dlopen's it with
  RTLD_NOW and exports nothing).
- `seed.py` — PutConfig/GetConfig driver (encodes the demo config protos, writes
  them to per).
- `snapshots/` — gitignored (live per state; regenerate with `tdb get-snapshot`).

## The flow (what was verified)

```
# 1. clean etcd + schema v1
ETCDCTL_API=3 etcdctl --endpoints=127.0.0.1:2379 del --prefix /theia/config/
artheia gen-schema apps/system/apps/component.art --out migration/schema_v1.json
theia install central && theia start central

# 2. seed v1 values, change some, snapshot
python migration/seed.py seed   --schema migration/schema_v1.json
python migration/seed.py change --schema migration/schema_v1.json
tdb get-snapshot snap_v1 --schema migration/schema_v1.json
theia stop central

# 3. add P4Config, evolve v1->v2, gen schema_v3, SCAFFOLD the transform
#    (edit apps/system/apps/package.art + demo.options, regen P4)
artheia gen-schema apps/system/apps/component.art --out migration/schema_v3.json
# gen-migration diffs the two schemas → per-node transform .json + BUILD entries
# (review the `_review` notes; set add-defaults; confirm renames):
artheia gen-migration --from migration/schema_v2.json --to migration/schema_v3.json \
    --out migration
theia start central
python migration/seed.py seed --schema migration/schema_v2.json   # seed P4Config v1

# 4. OFFLINE dev workflow: preview the migration on the host
tdb get-snapshot snap_p4v1 --schema migration/schema_v2.json
python tools/migrate/migrate.py \
    --snapshot migration/snapshots/snap_p4v1.json \
    --transform migration/p4config_v1_to_v2.json \
    --out migration/snapshots/snap_p4v2_offline.json

# 5. ONLINE: gen plugin, build, MigrateBulk on the live per
artheia gen-transform migration/p4config_v1_to_v2.json \
    --out migration/p4config_v1_to_v2.cc --schema migration/schema_v3.json
bazel build //migration:libper_migrate_p4config.so
# PerManager.MigrateBulk(P4Config, cfg_v1, cfg_v2, plugin_so=.../*.so)

# 6. COMPARE: online snapshot vs offline preview  (lockstep invariant)
tdb get-snapshot snap_p4v2_online --schema migration/schema_v3.json
diff <(jq .nodes.demo_fsm.config snap_p4v2_offline.json) \
     <(jq .nodes.demo_fsm.config snap_p4v2_online.json)   # → identical

# 7. change a v2 value, theia install + start, confirm it survives the restart.
```

## Bugs this verification surfaced (the seams)

The whole point — the individually-tested tools, joined for real, exposed three:

1. **per crashed on MigrateBulk** — `load_migration_plugin` read `dlerror()`
   TWICE in one expression (`x ? x : "unknown"`); dlerror clears on read, so the
   second call returned null and `std::string + null` segfaulted. Fixed (call
   once). (services/per/impl/migration_plugin.cc)
2. **the plugin .so wasn't self-contained** — it had undefined `pb_decode` /
   `system_apps_P4Config_msg` and a DT_NEEDED on the proto cc_library's .so, so
   per's RTLD_NOW dlopen failed. Fixed in this BUILD: compile demo.pb.c in +
   link nanopb statically + header-only proto dep (no runtime shared lib).
3. **gen-transform's `rename` codegen** emitted `to.label = from.tag` even though
   the plugin decodes both versions with the SAME (v2) nanopb struct (migration
   relies on field-NUMBER stability), where `from` has no `.tag`. A same-number
   rename is already carried; codegen now emits a no-op note. (artheia
   transform_codegen.py)
