model migrations as a **versioned DAG of document transformations** with operations `{moveKey, moveField, transform, split, merge}` plus an escape hatch `{custom}`. That is roughly the point where systems like Kubernetes CRD conversion,

Essential split: **schema evolution + data migration for PROTO documents stored in a KV store**. This problem has been solved in Kubernetes CRDs and Protobuf schema registries.

The key challenge is that you have:

1. **Storage schema** (paths/keys in etcd)
2. **Object schema** (PROTO structure)
3. **Versioned migrations**
4. **Cardinality changes** (1→N, N→1)
5. **Data transformations**
6. **Referential rewrites**

A pure declarative JSON transform language works well for ~80% of migrations, but some migrations inevitably require custom code.


---

Protobuf Evolution

Handled automatically.

```
add field
rename field
remove field (reserve tag)
default values
optional fields
```

---

Storage Migration DAG

Handled by your transform.json.

```
moveKey
split
merge
cross-document references
fan-out
path changes
payload reshaping
```

> **Theia scope:** of these, only **payload reshaping** is in scope for
> `services/per`. per is a config gatekeeper — **exactly one config per node**,
> at a fixed key `/theia/config/<node>` (one key = one object). So config
> migration is strictly **1:1**. The storage-TOPOLOGY ops (`moveKey`, `split`,
> `merge`, fan-out, cross-document refs) change key→object cardinality and have
> no home in per; the tooling **rejects** them
> (`migrate.validate_cardinality` errors on a topology op, `cardinality != 1:1`,
> or `maxFanout > 1`). A general document store with topology migration would be
> a *separate service*, not per. See the appendix.

These operate on decoded protobuf objects.

```
{
  "op": "split",
  "source": "/users/*",
  "outputs": [
    "/users/${id}",
    "/addresses/${id}"
  ]
}
```

---

Engine flow:


```
read protobuf blob
deserialize using old schema
apply migration DAG
serialize using current schema
write new keys
```

> **Theia: two engines, one rule-set.** The same rules run in two places that
> must agree:
> - **`tools/migrate/migrate.py`** — the architect's DESIGN/TEST bench. Operates
>   on the **decoded JSON** of a snapshot (`tdb get-snapshot`). JSON lives ONLY
>   here. This is where you record + eyeball a migration path before shipping.
> - **the dlopen'd plugin in `services/per`** (MigrateBulk, + lazy on read) —
>   the RUNTIME. Generated by `artheia gen-transform` and works DIRECTLY on the
>   **nanopb struct** (decode → struct-member ops → encode): **no JSON, no
>   libprotobuf** — it has to be fast. JSON in the hot path would be wrong.
>
> So: deserialize old (nanopb) → carry every field by default → apply the rules
> as `to.X = from.Y` / set / value-map → serialize current (nanopb) → re-put
> (CAS on rev). migrate.py and gen-transform are kept in **lockstep** on the
> rule semantics.

That separation keeps protobuf evolution simple and leverages what Protobuf already does well, while reserving your migration framework for structural changes that Protobuf cannot represent.

# Recommended Pattern

Introduce a new field.

```protobuf
message Address {
  string city = 1;
}

message User {
  string name = 1;

  string city = 2;       // old
  Address address = 3;   // new
}
```

Read logic:

```text
if address exists:
    use address.city
else:
    use city
```

After all data upgraded:

```protobuf
message User {
  string name = 1;
  Address address = 3;

  reserved 2;
}
```

This is exactly how large protobuf deployments evolve.

On ART level we adding message { field reserved } syntaxes to keep evidence for skipped proto id. Field newer deleted.

# More Formal Grammar

> **Theia versioning: content DIGEST, not version ints.** Instead of
> `"from": 1, "to": 2`, Theia keys each transform by a content-hash **digest**
> of the config shape (`cfg_<sha>`, from `artheia gen-schema`): `from_digest` →
> `to_digest`. The digest auto-detects "shape changed" with no human bump and
> can't drift; a stored value's digest ≠ the current schema digest is *exactly*
> when a migration is needed. Adjacent transforms chain by BFS over the edges
> (`v1→v2→v3` from `v1→v2` + `v2→v3`) — the "each step knows only N→N+1" model.
>
> **Field paths are JSONPath** (the dotted nested-object subset, `$.a.b`,
> jq-consistent — no wildcards/filters, which would imply topology). In the
> nanopb runtime a flat `$.x` is a struct member; a nested `$.a.b` that isn't a
> flat member is a signal to use a `custom` hook (below).

Something like:

```json
{
  "from": 1,
  "to": 2,
  "operations": [
    {
      "op": "moveKey"
    },
    {
      "op": "renameField"
    },
    {
      "op": "copyField"
    },
    {
      "op": "removeField"
    },
    {
      "op": "transform"
    },
    {
      "op": "split"
    },
    {
      "op": "merge"
    },
    {
      "op": "custom"
    }
  ]
}
```

## Split Object

Most important for cardinality changes.

Input:

```json
{
  "id": 1,
  "name": "john",
```

```text

```

Transform:

```json
{
  "op": "split",

  "outputs": [

        "name"

      "key": "/addresses/${id}",
      "fields": [
        "address"
      ]
    }

```

# What Cannot Be Fully Declarative

Some migrations are fundamentally procedural.

Example:

```text
Order
    ->
Invoice
```

where invoice totals must be recomputed from historical tax rules.

Or:

```text
User.address.country
```

must be looked up from an external service.

A transform language cannot safely express arbitrary business logic.

For this reason most mature systems eventually add:

```json
{
  "op": "custom",
  "handler": "UserToInvoiceMigration"
}
```

and run code.

A practical rule:

* 90% of migrations should be declarative.
* 10% should allow code hooks.

> **Theia: the code hook is already the foundation.** Every transform IS C — the
> `transform.json → gen-transform → .cc → dlopen plugin` path. So the declarative
> rules are *sugar over* the code-hook layer, not the other way round. A
> `{"op": "custom", "fn": "my_reshape"}` rule makes gen-transform emit
> `extern "C" void my_reshape(const Cfg* in, Cfg* out);` in the main plugin (+ a
> call) and write a **WRITE-ONCE** sidecar `<name>_custom.cc` with the typed
> nanopb-struct stub for you to fill — never clobbered on regen. So 100% is
> expressible (hand-write the hook on the typed structs), ~90% generated.

---

# Additional Constraints Worth Encoding

Every operation should declare cardinality:

```json
{
  "op": "split",
  "cardinality": "1:N"
}
```

Possible values:

```text
1:1
1:N
N:1
N:N
```

This lets the migration engine:

* validate transformations
* estimate document growth
* detect accidental fan-out explosions

Example:

```json
{
  "maxFanout": 10
}
```

to prevent creating millions of documents accidentally.

# Appendix — Theia tooling reference

The Theia design decisions are woven into the sections above (scope = 1:1
payload only, digests over version ints, JSONPath, nanopb-struct runtime, the
plugin-is-the-custom-hook). This appendix is the quick reference.

## The 4-tool chain

```
.art (config <Msg>)                         ← architect evolves the shape
   │  artheia gen-schema  → schema.json      (config_type → digest + fields)
   ▼
tdb get-snapshot  → snapshot.json            (decoded, every config a JSON obj)
   │  migrate.py --transform t.json          DESIGN BENCH: preview snapshot_n+1
   ▼  (eyeball the result; iterate on t.json)
artheia gen-transform t.json --schema schema.json → plugin.cc (+ _custom.cc)
   │  build .so (linkshared, like services/per/migrations/example)
   ▼
services/per MigrateBulk(plugin.so)          RUNTIME: nanopb struct reshape,
   + lazy migration on GetConfig read         re-put CAS — no JSON, no libprotobuf
```

migrate.py (JSON, design) and the gen-transform plugin (nanopb, runtime) are
kept **in lockstep** on the rule semantics.

## Rule vocabulary (transform.json)

Header: `{config_type, from_digest, to_digest, rules:[…]}` (+ optional
`cardinality:"1:1"`, `maxFanout:1`).

| op | meaning |
|----|---------|
| `rename {from, to}` | move a field (`$.a.b` paths) |
| `add {field\|path, default}` | add if absent (no clobber) |
| `remove {field\|path}` | drop a field |
| `set {field\|path, value}` | overwrite a field |
| `copy {from, to}` | duplicate a field |
| `transform {path, map, default?}` | value/enum remap; unmapped → `default` or unchanged |
| `custom {fn}` | call a user hook `void fn(const Cfg* in, Cfg* out)` in the write-once `_custom.cc` sidecar |

To extend the vocabulary, edit `migrate.apply_rule` (Python reference) **and**
`transform_codegen._emit_rules` (nanopb codegen) **together** — they must agree.

## Status

**Implemented:** gen-schema, tdb get-snapshot, migrate.py (JSONPath + value-map +
1:1 cardinality guard), gen-transform (nanopb-struct runtime + custom sidecar),
the dlopen plugin model (MigrateBulk + lazy read), `reserved <tag>` ART syntax
(holds a deleted field's proto tag — body order is the tag, so a deleted field
becomes a `reserved` slot).

**Deferred:**
- **Nested-message struct walk in gen-transform** — a flat `$.x` is a struct
  member; a nested `$.a.b` over a sub-message isn't a flat member, so today it's
  a `custom`-hook case. Walking nested nanopb structs (+ `.options`-pinning
  nested strings) would let the declarative rules reach sub-messages directly.
- **`reserved`-aware tag stability checks** — warn if a schema change reuses or
  skips a reserved tag, or shifts a field a stored value still uses.
