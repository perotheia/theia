# Supervisor signal ownership — split SIGTERM (generic main) vs SIGCHLD (engine)

> **DONE (2026-06-04).** Landed in `019e7c2`. `runtime.cpp` blocks **SIGCHLD
> only** via `pthread_sigmask` (not `sigprocmask`) into the engine's signalfd;
> SIGTERM/SIGINT stay owned by the generic `main.cc`, so the supervisor exits
> cleanly on SIGTERM.

## Problem

The supervisor process hangs on SIGTERM instead of exiting. Sending SIGTERM to a
standalone `bazel-bin/platform/supervisor/main/supervisor`:

```
[supervisor] shutdown requested (signal=15)   # engine drained its tree
[supervisor] supervisor exiting
[supervisor_worker] engine loop exited (rc=0)  # worker thread returned
# ...but the PROCESS stays alive (main thread in hrtimer_nanosleep, forever)
```

The live rig masks this — the launcher SIGKILLs the process group — so it only
surfaces when the binary is driven directly (e.g. probe/tdb e2e).

## Ownership model (decided)

Responsibilities split **by signal**:

- **SIGTERM / SIGINT — process termination — owned by the GENERIC `main.cc`.**
  This is NOT supervisor-specific: every gen-app FC process must cleanly stop
  its nodes and exit on SIGTERM. The generated main already does this (see
  below); it is the per-FC termination owner.
- **SIGCHLD + fork/exec/waitpid/restart — owned by the supervisor ENGINE only.**
  The executor is the unique supervising concern. No other FC forks children.
- **Child shutdown protocol:** the executor (parent) sends SIGTERM to a child
  and waits on its pid; the child's OWN `main.cc` catches SIGTERM, stops its
  nodes, and exits. So the child side is just the generic per-FC requirement.

### Root cause

The supervisor is the ONLY FC that breaks the generic SIGTERM contract — and it
breaks it precisely because its engine **over-blocks** SIGTERM/SIGINT, stealing
them from the generic handler.

1. `platform/supervisor/impl/core/runtime.cpp:143-150` blocks
   SIGCHLD **+ SIGTERM + SIGINT** via `sigprocmask` and consumes them through a
   `signalfd`. It runs in the `Supervisor` constructor, which
   `GenRunnable::start()` (`GenRunnable.hh:77`) executes on a worker
   `std::thread` (`SupervisorWorker_handlers.cc:109`).
   - `sigprocmask` in a multithreaded process is POSIX **undefined behavior**
     (`pthread_sigmask` required); on Linux it affects only the calling thread.
   - Net effect: SIGTERM is sometimes swallowed by the worker signalfd, never
     reaching the generic main handler.

2. The generic `main.cc` SIGTERM path is therefore dead for the supervisor:
   `std::signal(SIGTERM, on_signal)` → `g_running=false` never happens, so
   `main`'s `while (g_running) sleep` loop (main.cc:124) spins forever and the
   process never reaches `mux.stop()/ctl.stop()/worker.stop()` + `return 0`.

### What the code ALREADY does correctly (do NOT touch)

- **Generic per-FC SIGTERM termination is already implemented and is identical
  across FCs.** `services/sm/main/main.cc` (and every generated main) has:
  `std::signal(SIGINT/SIGTERM, on_signal)` → `g_running=false` →
  `while(g_running) sleep` → `config_mux.stop()` → each node `.stop("signal")`
  → `return 0`. The supervisor's main.cc is the SAME template (both
  AUTO-GENERATED by `artheia gen-app --kind fc`). This is the SIGTERM owner; it
  stays.
- **Child mask reset post-fork** (runtime.cpp:466): the child does
  `sigprocmask(SIG_SETMASK, &empty)` + `setsid()`, so the child's own main.cc
  SIGTERM handler works. ✓ ("child main.cc stops nodes and exit").
- **Executor SIGTERM→waitpid child-stop protocol** (runtime.cpp:561-588):
  `killpg(SIGTERM)` then `waitpid` with timeout + SIGKILL escalation. ✓
  ("executor sends SIGTERM to child and waits for pid").
- **Single exec owner**: fork (:458) / execvp (:513) / every waitpid
  (:567/:577/:586/:926) run only on the engine loop thread.
- **`state_mu_` (recursive)** serializes SupervisorCtl's direct `ctl_*` (TipcMux
  thread) against the loop's reap/sample/emit — orthogonal to signals, stays.

## Fix — engine blocks ONLY SIGCHLD; leave SIGTERM/SIGINT to generic main

Minimal and supervisor-local. **No generated-main.cc change, no generator
change, no clobber risk** — the only edit is the engine's signal setup.

### 1. Engine blocks SIGCHLD only

`runtime.cpp:143-150`: drop SIGTERM + SIGINT from the blocked set and from the
signalfd; block + consume **SIGCHLD only**. Use `pthread_sigmask` (the engine
runs on a spawned thread; `sigprocmask` is UB here):

```cpp
    // Engine owns child reaping: block SIGCHLD process-wide and read it via
    // signalfd. SIGTERM/SIGINT are NOT ours — the generic main.cc handler owns
    // process termination (every FC). pthread_sigmask: we're on a worker thread.
    sigset_t mask; sigemptyset(&mask);
    sigaddset(&mask, SIGCHLD);
    pthread_sigmask(SIG_BLOCK, &mask, nullptr);
    signal_fd_ = signalfd(-1, &mask, SFD_CLOEXEC | SFD_NONBLOCK);
```

Remove the SIGTERM/SIGINT cases from the signalfd drain in the run loop
(runtime.cpp ~828-834) — they can no longer arrive on this fd. The loop still
exits via `shutdown_requested_`, which `request_shutdown()` sets + wakes
(`cmd_eventfd_`).

NOTE on the SIGCHLD block scope: blocking SIGCHLD only on the engine thread (via
`pthread_sigmask` post-spawn) is enough for signalfd to receive it IF no other
thread has SIGCHLD unblocked AND handling it. No other thread installs a SIGCHLD
disposition, so default-ignore + the engine's block is fine. (If flakiness
appears, hoist JUST the SIGCHLD block to `main()` top before thread spawn — but
keep SIGTERM/SIGINT unblocked there so the generic handler still fires.)

### 2. Generic main.cc shutdown drives the engine

No code change needed — verify the path. On SIGTERM the generic handler sets
`g_running=false`; the loop exits; `worker.stop("signal")` →
`SupervisorWorker::do_stop()` → `g_engine->request_shutdown()` wakes the engine
loop, which drains the tree (SIGTERM→waitpid each child) and returns; `worker`
joins; `main` returns 0. The supervisor now terminates exactly like every other
FC.

### 3. Keep everything else

Engine loop stays the sole exec + SIGCHLD owner; `state_mu_` stays; child
protocol stays.

## Things to avoid (carry into review)

- The engine must NOT block or consume SIGTERM/SIGINT — those belong to generic
  main. Only SIGCHLD is the engine's.
- `sigprocmask` is banned in this multithreaded process — `pthread_sigmask`
  only (the existing `sigprocmask` at :148 and the child-context one at :467;
  the child one is post-fork single-threaded so technically OK, but switch for
  consistency).
- Exactly one `waitpid` owner (the loop). One exec owner. Never fork from a
  worker/handler thread.
- No `std::signal`/`sigaction` async handler in the engine touching
  mutex/CV/malloc/printf — signalfd avoids it; don't reintroduce.

## Deferred — full SignalThread/ExecThread split

A dedicated SignalThread (SIGTERM/SIGINT → ShutdownCoordinator) + separate
ExecThread (SIGCHLD + fork/exec/waitpid behind epoll/eventfd/timerfd) is correct
for a from-scratch multi-domain supervisor, but is more than this code needs: it
splits the working single-owner engine loop and duplicates its command-queue +
select machinery. Revisit only if restart-backoff (timerfd) or multiple
independent exec domains arrive. Recorded so the option isn't lost.

## Verification (when implemented)

1. Standalone: launch `supervisor` with a minimal tree, `kill -TERM <pid>` →
   process EXITS rc 0 within the shutdown budget (no `hrtimer_nanosleep` hang),
   children gone, log shows `supervisor exiting` + clean `main` return. Compare
   against a non-supervisor FC (e.g. sm) under the same SIGTERM — identical
   shape now.
2. Programmatic `Stop` RPC (tdb/probe) → same clean exit (shared
   `request_shutdown` path).
3. SIGCHLD still reaped: crash a child under load → restart still fires (the
   block change must not break SIGCHLD delivery to the engine signalfd). This is
   the one regression risk — test explicitly.
4. Threads: `ps -L` during shutdown shows worker/ctl/mux joined; engine dtor
   closes signalfd + cmd_eventfd (runtime.cpp:168-171).
